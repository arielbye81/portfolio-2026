# Global AGENTS.md for Frontend / UI/UX projects

## Default operating rules

### 第一性原理
请始终使用第一性原理思考问题。
你不能默认我已经非常清楚自己的真实目标、边界条件、约束和实现路径。
面对任何需求时，你必须先回到原始问题本身，识别以下内容：

- 真实目标是什么
- 当前问题的核心约束是什么
- 哪些是本质问题，哪些只是表象
- 哪些假设已经被明确给出，哪些仍然未经确认
- 当前要做的是“分析问题”、“提出方案”还是“直接实施”

如果动机、目标、约束、验收标准、边界条件不清晰，不允许直接进入编码、重构或视觉调整。
此时应先停下来，用简洁明确的方式指出不清晰之处，并与我讨论确认后再继续。

### 沟通原则
- 不要假设需求中未明确说明的业务规则、交互规则或视觉规则。
- 不要把“常见做法”当作“本题必须做法”。
- 不要因为追求完整性而擅自扩展需求。
- 当存在多个可能解释时，优先暴露歧义，而不是自行拍板。
- 先明确问题，再给方案，再实施。

### 执行原则
- 在修改代码前，先阅读相关上下文，理解页面结构、组件关系、数据流、状态流、路由、输入输出和现有实现方式。
- 不允许在未理解上下文的情况下直接改代码或改 UI。
- 优先做最小必要改动，保持最短路径实现。
- 不要修改与当前任务无关的文件、配置、依赖和行为。
- 不要把“顺手优化”“顺手重构”“顺手统一风格”混入当前任务。
- 每次修改都必须能回答：为什么改、改了什么、影响了什么、如何验证。

## 代码与界面规范

### Frontend / UI/UX 默认规范
当你编写、修改、重构任何前端代码或界面实现时，必须遵守以下规则：

- 做界面（布局、样式、组件视觉与交互）时，参考 Cursor skill `frontend-design`（`~/.cursor/skills/frontend-design/`），并在不违背本文件其他约束的前提下运用其设计取向。
- 代码必须清晰、可读、可维护，优先保证正确性，其次考虑简洁、可用性与性能。
- 不允许写“看起来能跑”但状态、交互、边界条件未经验证的代码。
- 不允许引入不必要的抽象层。
- 不允许为了“未来可能扩展”而预埋复杂结构。
- 不允许使用含糊命名，例如 `tmp`、`data2`、`newComponent`、`handleIt`。
- 组件职责必须单一，props、state、副作用、事件流必须明确。
- 对服务端组件 / 客户端组件边界、异步数据、渲染时机、缓存与 hydration 风险要显式关注。
- 对表单、列表、条件渲染、异步加载、错误态、空态、禁用态、hover / focus / active 状态要完整检查。
- 对响应式布局、间距系统、排版层级、视觉对齐、点击热区、信息层级要显式关注。
- 对可访问性要特别小心，例如：
  - 语义化标签是否正确
  - 图片是否有合理的 `alt`
  - 表单控件是否有关联 label
  - 键盘可达性是否成立
  - focus 状态是否可见
  - 颜色对比度是否足够
  - 仅靠颜色传达信息的情况是否被避免
- 任何涉及前端实现的修改，必须检查以下事项：
  - 数据流、状态流、事件流是否全链路一致
  - 服务端渲染、客户端渲染与 hydration 行为是否合理
  - 路由跳转、参数读取、加载态、错误态是否符合预期
  - DOM 结构和语义是否合理
  - 响应式行为是否在不同断点下成立
  - 交互反馈是否清晰且一致
  - 是否引入了隐式性能问题、重渲染问题或布局抖动

### TypeScript、注释与结构
- 优先为公开组件、公共函数、复杂 props、关键数据结构补充清晰的类型标注。
- 注释只写必要信息，重点解释“为什么”，而不是逐行翻译“做了什么”。
- 当逻辑涉及条件分支、状态同步、缓存、异步时序、布局约束、无障碍语义时，应写清楚。
- 不要添加噪音注释和过度注释。
- 保持 import、模块边界、组件层级、hooks、utils 组织清晰。

### React / Next.js 特殊要求
对于组件、页面、布局、hooks、表单、数据获取、交互逻辑，必须额外遵守：

- 明确区分展示组件、容器组件、hooks、工具函数职责，不要混写。
- 明确区分静态内容、客户端交互、异步数据逻辑，不要在一个组件里无边界堆叠。
- 不允许把实验性 hack 直接混入主逻辑而不标注。
- 不允许 silently 改变默认文案、交互语义、信息层级、路由语义或埋点语义。
- 修改交互逻辑时，必须检查：
  - 现有事件是否仍然正确触发
  - loading / success / error / empty 状态是否仍然闭合
  - 表单校验、提交、重置逻辑是否受影响
  - 依赖 effect 是否引入重复执行或竞态
  - memoization 是否真的必要且不改变语义
- 修改 UI 表现时，必须检查：
  - 视觉层级是否更清晰
  - 间距、字号、颜色、对齐是否仍然统一
  - 不同屏宽下是否仍然成立
  - 可点击区域和交互反馈是否合理
  - UI 改动是否改变了信息理解路径

### UI/UX 特殊要求
- 不要只把任务理解为“把页面改好看”。
- 必须先确认目标是提升可读性、可用性、转化、信息层级、品牌一致性还是交互效率。
- 不允许在没有明确目标的情况下随意改颜色、字号、间距、动效或布局。
- 不允许为了视觉统一而破坏业务优先级和操作效率。
- 不允许只考虑静态稿，不考虑真实内容长度、空数据、异常数据和响应式情况。
- 设计相关修改必须检查：
  - 用户是否更容易理解当前页面的主目标
  - 主次信息是否分层清晰
  - 关键操作是否更容易发现和执行
  - 文案、图标、颜色、反馈是否表达一致
  - 是否兼顾真实使用场景而不是仅在理想截图下成立

## 方案规范

当你需要给出修改方案、重构方案、交互方案、视觉方案时，必须遵守以下要求：

### 绝对约束
- 不允许给出兼容性或补丁性的方案。
- 不允许给出“先这样凑合、以后再优化”的方案。
- 不允许过度设计。
- 不允许自行引入我未提出的额外目标。
- 不允许擅自加入兜底、降级、双轨并行、兼容层、过渡层等方案，除非我明确要求。
- 不允许为了追求稳妥而改变业务逻辑本意、交互本意或信息表达本意。
- 必须遵循最短路径实现，但前提是逻辑正确。
- 必须经过页面结构、数据流、状态流、交互流和视觉层级验证后再输出方案。

### 输出方案时必须做的事
你在输出方案前，必须先完成以下思考：

1. 原始问题到底是什么
2. 这个方案解决的是根因，还是只是表象
3. 有没有引入额外复杂度
4. 有没有偏离我明确提出的需求
5. 整个组件关系、数据流、状态流、交互流是否逻辑闭合
6. 改动后渲染、交互、响应式、可访问性、埋点等链路是否仍然成立
7. 是否存在边界条件、可用性、视觉一致性或性能上的漏洞

如果以上任一点不能确认，不允许把方案表述成“可直接执行的正确方案”。

### 方案输出格式
当给出方案时，优先按以下结构组织：

1. 问题本质
2. 当前实现中的根因
3. 最短路径修改方案
4. 改动影响范围
5. 全链路正确性检查
6. 风险点与需要确认的前提
7. 实施步骤

如果我明确要求你直接改代码，则先在心里完成上述验证，再实施。

## 前端特殊检查要求

### 关于状态、渲染与交互语义
- 对任何状态修改，都默认关注状态来源、状态归属、更新时机、派生关系与副作用。
- 对 `useEffect`、`useMemo`、`useCallback`、受控组件、异步请求、路由参数、条件渲染要格外谨慎。
- 涉及列表渲染、key、排序、过滤、分页、搜索、表单联动时，必须检查状态一致性。
- 涉及弹窗、抽屉、菜单、tooltip、popover、toast、tab、accordion 等交互组件时，必须检查打开关闭逻辑、焦点管理和键盘操作。
- 涉及图片、视频、图表、动画时，必须检查加载性能、占位策略和布局稳定性。

### 关于视觉与可访问性
- 对任何视觉修改，都默认关注层级、对齐、节奏、留白、对比和一致性。
- 对颜色、字号、圆角、阴影、边框、动效的修改，不允许脱离现有设计语言随意发挥。
- 涉及表单、按钮、链接、卡片、导航时，必须检查 hover、focus、active、disabled 状态是否完整。
- 涉及长文案、多语言、极端数据长度时，必须检查换行、截断、溢出和布局稳定性。
- 涉及移动端时，必须检查触控热区、手势冲突和首屏可读性。

### 关于性能
- 不要为性能牺牲逻辑正确性、可用性或可维护性。
- 只有在不改变语义的前提下，才做必要的性能优化。
- 进行性能相关修改时，必须说明优化点在哪里，以及为什么不会改变结果语义。
- 注意不必要的客户端组件化、重复渲染、重复请求、无意义 memo、过大图片、无意义动画和布局抖动。

### 关于依赖与工程改动
- 不要主动新增依赖。
- 不要主动升级库版本。
- 不要主动改动环境配置、构建脚本、CI、Docker、启动参数，除非任务明确要求。
- 不要把当前任务扩展为工程治理任务。

## 修改代码前的默认检查清单

在真正动手修改前端代码前，先确认：

- 我是否已经理解真实目标
- 我是否已经理解相关页面、组件和调用链
- 我是否已经知道输入输出契约与交互预期
- 我是否已经检查数据流 / 状态流 / 渲染边界 / 可访问性语义
- 我是否已经识别响应式、加载态、错误态、空态影响
- 我是否避免了补丁式和兼容式方案
- 我是否没有擅自扩展需求
- 我是否可以给出一个逻辑闭合的最短路径方案

如果以上任一项答案是否定的，先停下来讨论，不要直接编码。

---

## 修改代码后的默认检查清单

完成修改后，必须检查：

- 逻辑是否与原始需求严格一致
- 是否只改了必要内容
- 是否破坏了数据流、状态流、渲染边界或交互语义
- 是否破坏了响应式、可访问性和视觉一致性
- 是否引入了隐藏状态问题、竞态或性能问题
- 是否引入了不必要复杂度
- 是否存在边界条件遗漏
- 是否能清楚解释改动原因、影响范围和验证方法

## 回复风格
默认用简洁、直接、工程化的方式回复。
优先给出结论，再给关键依据。
不要堆砌空泛建议。
不要为了显得全面而输出未经验证的内容。
如果存在不确定性，明确指出不确定点和缺失前提。
